# 回滚方案文档

## 概述

本文档描述了文章内容类型默认值修复的回滚方案。如果在部署过程中发现严重问题，可以按照本文档进行回滚。

## 回滚触发条件

以下情况应考虑回滚：

1. **严重功能故障**：系统无法正常创建或更新文章
2. **数据完整性问题**：现有数据被错误修改
3. **性能严重下降**：系统响应时间显著增加
4. **兼容性问题**：现有 API 客户端无法正常工作
5. **用户投诉激增**：大量用户报告相同问题

## 快速回滚决策矩阵

| 问题类型 | 回滚时机 | 回滚优先级 |
|---------|----------|-----------|
| 系统无法启动 | 立即 | 🔴 紧急 |
| 数据丢失/损坏 | 立即 | 🔴 紧急 |
| 核心功能故障 | 30分钟内 | 🟠 高 |
| 性能严重下降 | 1小时内 | 🟠 高 |
| 非核心功能问题 | 计划维护 | 🟡 中 |
| 轻微用户体验问题 | 下次部署 | 🟢 低 |

## 回滚方案

### 方案 1：代码回滚

**适用情况**：代码变更导致的问题，数据库无问题

**回滚步骤**：

```bash
# 1. 查看最近的提交
git log --oneline -5

# 2. 回滚到之前的提交
git revert <commit_hash>

# 或使用 git reset（谨慎使用）
# git reset --hard <commit_hash>

# 3. 重新部署代码
bun run build
# 根据你的部署方式执行部署命令
```

**验证步骤**：

```bash
# 1. 检查代码版本
git log --oneline -1

# 2. 验证核心功能
# - 尝试创建新文章
# - 检查 API 响应正常
# - 确认数据库连接正常

# 3. 检查日志
tail -f logs/error.log
```

### 方案 2：数据库回滚

**适用情况**：数据库迁移导致的问题

**回滚步骤**：

```sql
-- 1. 停止应用服务，防止新数据写入

-- 2. 回滚数据库默认值变更
ALTER TABLE posts ALTER COLUMN content_type SET DEFAULT 'markdown';

-- 3. 验证回滚结果
SELECT
    COLUMN_NAME,
    COLUMN_DEFAULT
FROM
    INFORMATION_SCHEMA.COLUMNS
WHERE
    TABLE_SCHEMA = DATABASE()
    AND TABLE_NAME = 'posts'
    AND COLUMN_NAME = 'content_type';

-- 预期结果：COLUMN_DEFAULT 应该显示 'markdown'
```

**历史数据回滚**（如果执行了数据迁移）：

```bash
# 1. 从备份恢复整个数据库
mysql -u root -p blog < backup_20260307_143000.sql

# 或恢复压缩的备份
gunzip < backup_20260307_143000.sql.gz | mysql -u root -p blog

# 2. 验证数据完整性
mysql -u root -p blog -e "SELECT COUNT(*) FROM posts;"
mysql -u root -p blog -e "SELECT content_type, COUNT(*) FROM posts GROUP BY content_type;"
```

### 方案 3：完整回滚

**适用情况**：代码和数据库都有问题

**回滚步骤**：

1. **停止服务**

```bash
# 停止应用服务
# 根据你的部署方式执行
# 例如：systemctl stop blog-service
# 或：docker-compose down
```

2. **数据库回滚**

```bash
# 从备份恢复数据库
mysql -u root -p blog < backup_20260307_143000.sql
```

3. **代码回滚**

```bash
# 回滚代码到之前版本
git revert <commit_hash>
# 或
git reset --hard <commit_hash>
```

4. **重新部署**

```bash
# 构建和部署
bun run build
# 执行部署命令
```

5. **启动服务**

```bash
# 启动应用服务
# 例如：systemctl start blog-service
# 或：docker-compose up -d
```

6. **验证**

```bash
# 检查服务状态
# 检查日志
# 测试核心功能
```

## 回滚脚本模板

```bash
#!/bin/bash
# 回滚脚本模板
# 用法：./rollback.sh <方案编号>

ROLLBACK_TYPE=$1
BACKUP_FILE="backup_20260307_143000.sql"

case $ROLLBACK_TYPE in
  1)
    echo "执行方案 1：代码回滚"
    git revert HEAD
    # 执行部署命令
    ;;
  2)
    echo "执行方案 2：数据库回滚"
    mysql -u root -p blog < $BACKUP_FILE
    ;;
  3)
    echo "执行方案 3：完整回滚"
    # 停止服务
    # 回滚数据库
    mysql -u root -p blog < $BACKUP_FILE
    # 回滚代码
    git reset --hard <commit_hash>
    # 重新部署
    # 启动服务
    ;;
  *)
    echo "错误：未知回滚方案"
    echo "用法：./rollback.sh <1|2|3>"
    exit 1
    ;;
esac

echo "回滚完成，请执行验证步骤"
```

## 注意事项

⚠️ **重要提醒**：

1. 回滚是不可逆的操作，请谨慎决策
2. 回滚前务必备份当前状态
3. 紧急情况下，优先恢复服务正常运行
4. 回滚后必须进行完整的功能验证
5. 记录回滚过程和结果
6. 及时通知相关方回滚状态

## 回滚决策树

```
发现问题
    ↓
评估影响
    ↓
影响严重？├─ 是 → 执行紧急回滚
    └─ 否 → 继续评估
        ↓
    能快速修复？
        ├─ 是 → 尝试修复
        └─ 否 → 执行计划回滚
            ↓
    选择回滚方案
        ↓
    执行回滚
        ↓
    验证结果
        ↓
    回滚成功？├─ 是 → 分析问题，制定修复计划
        └─ 否 → 升级处理，联系高层
```

## 持续改进

每次回滚后，都应该：

1. 记录回滚的经验教训
2. 更新回滚文档和脚本
3. 改进部署和监控流程
4. 加强测试覆盖率
5. 优化回滚决策流程

通过持续的改进，减少未来需要回滚的概率。
