## 新增需求

### 需求:文章编辑页面必须支持编辑器类型选择
文章编辑页面必须允许用户选择文章的编辑器类型（Markdown 或富文本），并在选择后锁定该类型。

#### 场景:新建文章时选择编辑器类型
- **当** 用户创建新文章
- **那么** 编辑页面必须显示编辑器类型选择器（Markdown / 富文本）
- **当** 用户选择编辑器类型
- **那么** 编辑页面必须显示对应的编辑器
- **那么** 编辑器类型选择器必须禁用，不允许再修改

#### 场景:编辑已有文章时显示对应编辑器
- **当** 用户编辑已有文章
- **那么** 编辑页面必须根据文章的 content_type 显示对应的编辑器
- **那么** 编辑器类型选择器必须禁用，不允许修改类型
- **当** 文章的 content_type 为空或 null
- **那么** 系统必须默认显示富文本编辑器

### 需求:文章列表页面必须显示文章类型标识
文章列表页面必须在列表中显示每篇文章的编辑器类型标识。

#### 场景:列表页显示类型列
- **当** 用户查看文章列表
- **那么** 列表必须包含"类型"列
- **当** 文章类型为 markdown
- **那么** 类型列必须显示 Markdown 标识（图标或标签）
- **当** 文章类型为 rich-text
- **那么** 类型列必须显示富文本标识（图标或标签）

### 需求:文章保存必须包含编辑器类型
系统在保存文章时必须将编辑器类型（content_type）一起提交到后端。

#### 场景:保存新建文章
- **当** 用户点击保存按钮
- **那么** 系统必须将 title、content、tagIds、visible、content_type 一起提交到后端

#### 场景:保存编辑文章
- **当** 用户保存编辑后的文章
- **那么** 系统必须保持原有的 content_type 不变
- **那么** 系统必须将所有字段（包括 content_type）一起提交到后端

## 修改需求

### 需求:前端类型定义必须包含 content_type 字段
前端 API 类型定义必须包含 content_type 字段以支持文章类型区分。

#### 场景:文章详情数据包含 content_type
- **当** 系统从后端获取文章详情
- **那么** PostDetailData 类型必须包含 content_type 字段

#### 场景:文章列表项包含 content_type
- **当** 系统从后端获取文章列表
- **那么** PostListItem 类型必须包含 content_type 字段

#### 场景:文章提交数据包含 content_type
- **当** 系统提交文章数据到后端
- **那么** PostPatchData 类型必须包含 content_type 字段
