## 上下文

当前博客系统后端已实现文章类型支持（markdown 和 rich-text），但前端只实现了富文本编辑器（基于 WangEditor）。前端项目已安装 `md-editor-v3` 和 `highlight.js` 依赖，但没有封装成可用的组件。

**当前状态：**
- 后端：`PostEntity` 有 `content_type` 字段，默认值为 'markdown'；`CreatePostDto` 接受 `content_type` 参数（markdown/rich-text）
- 前端：只有 `ArtWangEditor` 组件；文章编辑页面只支持富文本编辑；类型定义缺少 `content_type` 字段

**约束：**
- 一旦选择了编辑器类型，编辑阶段不允许修改（简化实现，避免数据转换复杂性）
- 必须保持与现有 ArtWangEditor 组件的 API 一致性
- 需要支持旧数据兼容（content_type 为空/null 的情况）

## 目标 / 非目标

**目标：**
- 提供完整的 Markdown 编辑器组件，支持基础编辑、预览、代码高亮、图片上传
- 在文章编辑页面实现编辑器类型选择功能（新建时可选，编辑时锁定）
- 在文章列表页面显示文章类型标识
- 更新前端类型定义以支持 content_type 字段

**非目标：**
- 不支持编辑器类型切换时的数据格式转换（Markdown ↔ HTML）
- 不支持在编辑阶段修改编辑器类型
- 不支持复杂的 Markdown 扩展功能（如数学公式、图表等）

## 决策

### 1. Markdown 编辑器库选择

**决策：** 使用 `md-editor-v3` 作为 Markdown 编辑器

**理由：**
- 项目已安装该库（v4.17.0）
- 提供完整的编辑和预览功能
- 支持代码高亮、图片上传等基础功能
- 有良好的 Vue 3 集成
- 社区活跃，文档完善

**替代方案：**
- `vue-markdown-editor`：功能相对简单
- `toast-ui.editor`：功能强大但包体积较大
- 自研编辑器：开发成本高，维护复杂

### 2. 编辑器组件封装

**决策：** 创建 `ArtMarkdownEditor` 组件，封装 `md-editor-v3`，保持与 `ArtWangEditor` 一致的 API

**理由：**
- 统一组件接口，便于切换和维护
- 参考现有 ArtWangEditor 的封装模式
- 统一配置样式和行为

**组件 API 设计：**
```typescript
interface Props {
  /** 编辑器高度 */
  height?: string
  /** 编辑器模式：preview/edit/live */
  mode?: 'preview' | 'edit' | 'live'
  /** 占位符文本 */
  placeholder?: string
  /** 是否禁用 */
  disabled?: boolean
}
```

**暴露方法：**
- `getEditor()`: 获取编辑器实例
- `setContent(text)`: 设置内容
- `getContent()`: 获取内容
- `focus()`: 聚焦编辑器

### 3. 编辑器类型选择方案

**决策：** 在编辑页面内实现编辑器类型选择（方案 B）

**理由：**
- 流程流畅，用户体验好
- 无需额外的中间页面
- 逻辑集中在编辑页面，易于维护

**实现逻辑：**
- 新建文章：显示编辑器类型选择器，用户选择后锁定并显示对应编辑器
- 编辑已有文章：根据 content_type 固定显示对应编辑器，禁用类型选择器
- 默认类型：content_type 为空/null 时，默认显示富文本编辑器

**替代方案：**
- 方案 A（独立选择页面）：多一个步骤，用户体验较差
- 方案 C（URL 参数）：需要修改列表页，增加复杂度

### 4. 编辑器类型标识显示

**决策：** 在列表页添加单独的"类型"列，使用 Element Plus Tag 组件显示

**理由：**
- 清晰直观，用户一眼就能看出文章类型
- 使用现有 UI 组件，保持样式一致

**显示规则：**
- Markdown：蓝色 Tag，显示 "MD"
- Rich-text：绿色 Tag，显示 "富文本"

### 5. 旧数据处理

**决策：** content_type 为空/null 的文章，前端默认使用富文本编辑器

**理由：**
- 富文本编辑器是现有功能，向后兼容
- 用户可以继续使用已有文章
- 即使后端默认值是 'markdown'，前端可以选择不同的默认行为

### 6. 图片上传配置

**决策：** Markdown 编辑器使用与 WangEditor 相同的上传接口和配置

**理由：**
- 统一上传逻辑，便于维护
- 复用现有的后端接口

**上传配置：**
- 接口：`/api/common/upload/wangeditor`
- 最大文件大小：3MB
- 支持的文件类型：image/*
- 携带 Authorization header

## 风险 / 权衡

### 风险 1：旧数据兼容性
**风险：** 现有数据库中的文章可能没有 content_type 字段或字段为空
**缓解措施：** 前端进行空值判断，默认显示富文本编辑器

### 风险 2：编辑器性能
**风险：** md-editor-v3 在处理大文件时可能会有性能问题
**缓解措施：** 设置合理的内容长度限制，监控性能表现

### 风险 3：组件 API 一致性
**风险：** ArtMarkdownEditor 与 ArtWangEditor 的 API 可能不完全一致
**缓解措施：** 定义清晰的接口规范，确保关键方法和 props 保持一致

### 风险 4：用户体验不一致
**风险：** 两个编辑器的交互方式、快捷键等可能不同
**缓解措施：** 提供清晰的用户提示，帮助文档说明不同编辑器的特性

### 权衡 1：功能完整性 vs 实现复杂度
**决策：** 只实现基础的 Markdown 编辑功能，不包含高级扩展
**理由：** 降低实现复杂度，快速上线，后续可根据需求迭代

### 权衡 2：编辑器类型切换 vs 数据一致性
**决策：** 不支持编辑器类型切换
**理由：** 避免数据格式转换的复杂性，降低出错风险
