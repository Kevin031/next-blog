## 新增需求

### 需求:文章编辑页面必须采用分层布局结构
文章编辑页面必须采用四层布局结构，从上到下依次为顶部固定栏、工具栏、编辑器区域和底部状态栏，以提高空间利用率和编辑体验。

#### 场景:顶部固定栏包含标题和操作按钮
- **当** 用户访问文章编辑页面
- **那么** 顶部固定栏必须显示返回按钮、标题输入框和保存按钮
- **当** 页面滚动时
- **那么** 顶部固定栏必须保持固定可见

#### 场景:工具栏包含标签和编辑器类型选择
- **当** 用户访问文章编辑页面
- **那么** 工具栏必须显示标签选择器和编辑器类型选择器
- **当** 在桌面模式下
- **那么** 工具栏必须采用左右布局，标签选择器在左，编辑器类型选择器在右

#### 场景:编辑器区域使用自适应高度
- **当** 用户访问文章编辑页面
- **那么** 编辑器区域必须使用最小高度 600px
- **当** 浏览器窗口大小变化时
- **那么** 编辑器区域必须自动调整高度以适应可用空间

#### 场景:底部状态栏显示发布状态
- **当** 用户访问文章编辑页面
- **那么** 底部状态栏必须显示文章的发布状态（已发布/草稿）
- **当** 用户点击发布状态切换
- **那么** 系统必须切换文章的可见性状态

### 需求:文章编辑页面必须支持快捷键保存
文章编辑页面必须支持 Ctrl+S / Cmd+S 快捷键保存文章，以提高编辑效率。

#### 场景:使用快捷键保存文章
- **当** 用户在编辑页面按下 Ctrl+S 或 Cmd+S
- **那么** 系统必须触发保存操作
- **当** 保存操作成功
- **那么** 系统必须显示保存成功提示

#### 场景:快捷键保存时显示按钮反馈
- **当** 用户在编辑页面按下 Ctrl+S 或 Cmd+S
- **那么** 保存按钮必须显示点击动画或高亮效果
- **当** 保存操作完成
- **那么** 保存按钮必须恢复正常状态

### 需求:标题输入框必须显示字符计数
文章编辑页面的标题输入框必须实时显示字符计数，帮助用户控制标题长度。

#### 场景:显示字符计数
- **当** 用户在标题输入框中输入内容
- **那么** 输入框必须显示当前字符数和最大字符数（例如：50/100）
- **当** 字符数未超过最大限制
- **那么** 字符计数必须以默认颜色显示

#### 场景:超过最大字符数时提示
- **当** 用户在标题输入框中输入超过 100 个字符
- **那么** 字符计数必须以警告颜色显示（如红色）
- **当** 字符数超过最大限制
- **那么** 输入框必须阻止进一步输入或显示警告提示

### 需求:文章编辑页面必须支持全屏编辑模式
文章编辑页面必须支持全屏编辑模式，允许用户专注于内容创作而不受界面元素干扰。

#### 场景:进入全屏模式
- **当** 用户点击全屏按钮
- **那么** 系统必须隐藏顶部固定栏和工具栏
- **那么** 编辑器区域必须占满整个屏幕
- **那么** 必须显示退出全屏按钮

#### 场景:退出全屏模式
- **当** 用户点击退出全屏按钮
- **当** 用户在全屏模式下按下 Esc 键
- **那么** 系统必须恢复顶部固定栏和工具栏
- **那么** 编辑器区域必须恢复到正常大小

#### 场景:全屏模式下保持数据同步
- **当** 用户在全屏模式下编辑内容
- **那么** 编辑器内容必须与表单数据保持同步
- **当** 用户退出全屏模式
- **那么** 编辑器内容必须保持在退出前的状态

### 需求:文章编辑页面必须支持响应式设计
文章编辑页面必须根据设备屏幕尺寸自动调整布局，提供良好的多设备体验。

#### 场景:桌面模式显示完整布局
- **当** 用户在桌面设备（宽度 > 1200px）访问编辑页面
- **那么** 系统必须显示完整布局，包括所有工具栏和状态栏
- **那么** 编辑器区域必须充分利用屏幕宽度

#### 场景:平板模式显示压缩布局
- **当** 用户在平板设备（宽度 768px - 1200px）访问编辑页面
- **那么** 系统必须显示压缩布局，简化工具栏
- **那么** 编辑器类型选择器必须采用紧凑样式

#### 场景:移动模式显示简化布局
- **当** 用户在移动设备（宽度 < 768px）访问编辑页面
- **那么** 系统必须显示垂直堆叠布局
- **那么** 工具栏元素必须分行显示
- **那么** 编辑器必须使用固定高度（400px）

### 需求:文章编辑页面必须预留扩展接口
文章编辑页面的工具栏和状态栏必须提供插槽接口，支持未来插入新组件而无需重构布局。

#### 场景:工具栏提供中间插槽
- **当** 开发者需要在工具栏中间添加新组件
- **那么** 工具栏必须提供命名的中间插槽
- **当** 组件插入到中间插槽
- **那么** 组件必须显示在标签选择器和编辑器类型选择器之间

#### 场景:状态栏提供中间插槽
- **当** 开发者需要在状态栏中间添加新组件
- **那么** 状态栏必须提供命名的中间插槽
- **当** 组件插入到中间插槽
- **那么** 组件必须显示在发布状态和快捷键提示之间

## 修改需求

### 需求:文章编辑页面必须支持编辑器类型选择
文章编辑页面必须允许用户选择文章的编辑器类型（Markdown 或富文本），并在选择后锁定该类型。编辑器类型选择器必须位于工具栏的右侧区域。

#### 场景:编辑器类型选择器位于工具栏右侧
- **当** 用户访问文章编辑页面
- **那么** 工具栏右侧必须显示编辑器类型选择器
- **当** 用户点击编辑器类型选择器
- **那么** 系统必须显示 Markdown 和富文本选项的单选按钮

#### 场景:新建文章时选择编辑器类型
- **当** 用户创建新文章
- **那么** 编辑页面工具栏必须显示编辑器类型选择器（Markdown / 富文本）
- **当** 用户选择编辑器类型
- **那么** 编辑页面必须显示对应的编辑器
- **那么** 编辑器类型选择器必须禁用，不允许再修改

#### 场景:编辑已有文章时显示对应编辑器
- **当** 用户编辑已有文章
- **那么** 编辑页面必须根据文章的 content_type 显示对应的编辑器
- **那么** 编辑器类型选择器必须禁用，不允许修改类型
- **当** 文章的 content_type 为空或 null
- **那么** 系统必须默认显示富文本编辑器
