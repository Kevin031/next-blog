# 设计：修复文章类型默认值

## 上下文

### 当前状态
文章系统存在数据一致性问题：
- 前端 `edit.vue` 中 `formData.content_type` 默认值已正确设置为 `'rich-text'`
- 后端 `PostEntity` 数据库实体中 `content_type` 字段默认值错误设置为 `'markdown'`
- 后端 `CreatePostDto` API 文档中 `content_type` 字段默认值错误设置为 `'markdown'`
- 导致部分数据在创建或更新过程中被降级为 `markdown` 类型

### 约束条件
- 必须保持向后兼容，`content_type` 字段是可选的
- 不能破坏现有的 API 接口
- 需要考虑历史数据的处理
- 前端编辑器已经正确默认为富文本编辑器

### 利益相关者
- 开发者：需要确保 API 和数据库的一致性
- 管理员：需要处理历史错误数据
- 用户：期望文章默认使用富文本编辑器

## 目标 / 非目标

**目标：**
- 统一文章内容类型的默认值为 `'rich-text'`
- 确保前端、API、数据库三个层面的默认值一致
- 提供数据迁移机制来修正历史错误数据
- 保持 API 向后兼容

**非目标：**
- 修改前端编辑器的默认行为（已经是正确的）
- 重构整个文章管理系统
- 修改内容类型验证逻辑（保持现有验证）

## 决策

### 1. 数据库实体默认值修改
**决策：** 修改 `PostEntity` 中 `content_type` 字段的 `@Column` 装饰器 `default` 值从 `'markdown'` 改为 `'rich-text'`

**理由：**
- TypeORM 使用实体定义的默认值作为数据库层面的默认值
- 这确保了即使前端未传递该字段，数据库也会使用正确的默认值
- 符合业务逻辑期望

**替代方案考虑：**
- 仅在 Service 层处理默认值：不够彻底，可能绕过 Service 的直接数据库操作仍会使用错误的默认值
- 使用数据库触发器：过度设计，增加复杂性

### 2. API DTO 默认值修改
**决策：** 修改 `CreatePostDto` 中 `@ApiProperty` 装饰器的 `default` 值从 `'markdown'` 改为 `'rich-text'`

**理由：**
- Swagger API 文档会显示正确的默认值
- 为开发者提供准确的 API 使用参考
- 与业务期望一致

**替代方案考虑：**
- 不修改 API 文档默认值：会导致文档与实际行为不一致，造成混淆

### 3. DTO 运行时默认值处理
**决策：** 在 `CreatePostDto` 中为 `content_type` 字段添加运行时默认值处理

**理由：**
- 确保即使前端传递 `undefined` 或 `null`，DTO 验证和转换过程中也会使用正确的默认值
- 提供多层保护，防止意外的空值导致 TypeORM 使用实体默认值
- 增强数据一致性的保障

**实现方式：**
```typescript
@ApiProperty({ description: '内容格式类型: markdown 或 rich-text', required: false, enum: ['markdown', 'rich-text'], default: 'rich-text' })
@IsOptional()
@IsIn(['markdown', 'rich-text'], { message: '内容格式类型必须是 markdown 或 rich-text' })
readonly content_type: string = 'rich-text'; // 添加运行时默认值
```

**替代方案考虑：**
- 仅依赖数据库默认值：可能在某些边界情况下失效
- 在 Controller 层处理：逻辑分散，不统一

### 4. 历史数据处理策略
**决策：** 提供可选的数据迁移脚本，但不强制执行

**理由：**
- 某些历史数据可能是用户明确选择 `markdown` 类型的，不应该强制修改
- 提供脚本让管理员根据实际情况决定是否执行
- 脚本可以基于启发式规则判断（如内容格式、HTML 标签等）

**迁移策略：**
```sql
-- 仅更新明显错误的数据：content_type 为 'markdown' 但内容包含 HTML 标签
UPDATE posts
SET content_type = 'rich-text'
WHERE content_type = 'markdown'
  AND content LIKE '%<%>'
  AND content LIKE '%>%';
```

**替代方案考虑：**
- 强制批量更新所有历史数据：风险过高，可能覆盖用户明确的选择
- 不处理历史数据：问题持续存在，影响数据质量

## 风险 / 权衡

### 风险 1：现有 API 依赖行为
**风险：** 如果有客户端代码依赖 `'markdown'` 默认值，可能会导致意外行为

**缓解措施：**
- `content_type` 字段是可选的，显式传递的值不受影响
- 主要影响新创建的文章，现有文章不会被自动修改
- 变更符合用户的实际期望和前端行为

### 风险 2：历史数据误判
**风险：** 数据迁移脚本可能错误地将某些确实是 markdown 格式的文章识别为富文本

**缓解措施：**
- 提供基于启发式规则的谨慎迁移脚本
- 建议管理员先备份数据库再执行迁移
- 提供回滚机制或确认步骤

### 风险 3：TypeORM 同步行为
**风险：** 在生产环境中 `synchronize: true` 可能导致意外的表结构变更

**缓解措施：**
- 建议生产环境关闭 `synchronize: true`
- 使用明确的数据库迁移脚本管理表结构变更
- 在测试环境充分验证后再部署

### 权衡 1：简单性与完整性
- 当前方案专注于修复默认值问题，保持简单
- 没有重构整个文章类型系统，避免过度设计

### 权衡 2：自动化与人工干预
- 迁移脚本需要管理员主动执行，而非自动运行
- 这减少了对现有数据的潜在影响

## 迁移计划

### 部署步骤

1. **代码变更**
   - 修改 `server/src/posts/entities/post.entity.ts` 中的默认值
   - 修改 `server/src/posts/dto/create-post.dto.ts` 中的 API 文档和运行时默认值
   - 运行测试确保变更正常工作

2. **数据库更新**
   - 在开发环境测试 `synchronize: true` 是否正确更新表结构
   - 生产环境建议使用 SQL 命令手动更新：
     ```sql
     ALTER TABLE posts ALTER COLUMN content_type SET DEFAULT 'rich-text';
     ```

3. **可选数据迁移**
   - 备份数据库
   - 运行提供的数据迁移脚本（如管理员选择）
   - 验证迁移结果

### 回滚策略

- 代码回滚：简单的 git revert
- 数据库回滚：如果手动修改了表结构，可以恢复到之前的默认值
- 数据回滚：如果执行了迁移脚本，从备份恢复
