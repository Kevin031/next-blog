## 新增需求

### 需求:文章必须包含内容格式类型字段
系统必须为每篇文章存储内容格式类型，以标识内容是富文本还是 Markdown 格式。内容格式类型字段必须为 `content_type`，类型为字符串。

#### 场景:创建文章时指定格式类型
- **当** 用户创建新文章并在请求中指定 `content_type` 为 `markdown` 或 `rich-text`
- **那么** 系统必须将此格式类型保存到数据库的 `content_type` 字段

#### 场景:创建文章时未指定格式类型
- **当** 用户创建新文章但未在请求中提供 `content_type` 字段
- **那么** 系统必须使用默认值 `markdown` 作为内容格式类型

### 需求:内容格式类型必须进行数据验证
系统必须验证 `content_type` 字段的值，确保只接受合法的格式类型。

#### 场景:验证合法的格式类型
- **当** 用户在创建或更新文章时提供 `content_type` 为 `markdown` 或 `rich-text`
- **那么** 系统必须接受该请求并保存数据

#### 场景:验证非法的格式类型
- **当** 用户在创建或更新文章时提供 `content_type` 为 `markdown` 或 `rich-text` 之外的值
- **那么** 系统必须拒绝该请求并返回验证错误信息

### 需求:文章详情接口必须返回内容格式类型
系统必须在获取文章详情的响应中包含 `content_type` 字段，以便前端能够正确渲染内容。

#### 场景:获取文章详情
- **当** 用户请求文章详情接口（GET /posts/:id）
- **那么** 系统必须在响应中包含 `content_type` 字段，其值为该文章存储的格式类型

### 需求:文章列表接口必须返回内容格式类型
系统必须在获取文章列表的响应中包含每篇文章的 `content_type` 字段，以便前端能够在列表中正确渲染内容预览。

#### 场景:获取文章列表
- **当** 用户请求文章列表接口（GET /posts）
- **那么** 系统必须在响应的每篇文章中包含 `content_type` 字段，其值为该文章存储的格式类型

### 需求:更新文章时必须允许修改内容格式类型
系统必须允许用户在更新文章时修改内容格式类型。

#### 场景:更新文章的格式类型
- **当** 用户更新文章并修改 `content_type` 字段
- **那么** 系统必须更新数据库中的 `content_type` 字段为新值
