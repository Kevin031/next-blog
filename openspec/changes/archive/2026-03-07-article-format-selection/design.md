## 上下文

当前博客系统的文章内容存储在单一的 `content` 字段中，没有区分内容格式。系统使用 TypeORM + MySQL，文章实体（PostEntity）包含标题、作者、内容、可见性等字段。前端使用 Vue 3 + Element Plus，后端使用 NestJS 提供 API 接口。

现有系统已经具备文章的 CRUD 功能，包括创建、更新、删除和查询文章。但在内容格式方面缺乏灵活性，用户无法选择使用富文本或 Markdown 编辑器。

## 目标 / 非目标

**目标：**
- 在数据库中添加 `content_type` 字段，支持存储 `rich-text` 或 `markdown` 两种格式
- 在 API 接口（创建、更新、获取）中添加和返回 `content_type` 字段
- 添加数据验证，确保 `content_type` 只接受有效的格式值
- 为现有数据提供默认值（markdown）

**非目标：**
- 不修改前端编辑器 UI（前端适配属于独立任务）
- 不实现内容格式转换（富文本和 Markdown 之间不互相转换）
- 不修改文章列表查询逻辑（通过 PostEntity 自动包含 content_type 字段）

## 决策

### 数据模型设计

**决策：** 在 PostEntity 中添加 `content_type` 列，使用 VARCHAR(20) 类型

**理由：**
- VARCHAR(20) 足够存储 'rich-text' 和 'markdown'，预留空间用于未来扩展
- 与现有的 `title`（VARCHAR(50)）和 `author`（VARCHAR(20)）字段风格保持一致
- 避免使用 ENUM 类型，保持数据库灵活性

**替代方案：**
- ENUM 类型：虽然能保证数据一致性，但修改选项需要 ALTER TABLE，灵活性较差
- 整型常量（如 0=markdown, 1=rich-text）：代码可读性差，不利于数据库查询和调试

### 默认值策略

**决策：** 新增字段默认值为 'markdown'

**理由：**
- Markdown 是技术博客的常见格式
- 现有内容大多数可能是纯文本或 Markdown
- 向后兼容，不破坏现有功能

**替代方案：**
- 默认值为 'rich-text'：不合适，现有内容可能不是富文本格式
- 允许 NULL：会导致前端需要处理空值逻辑，增加复杂度

### API 字段设计

**决策：** 在所有相关 DTO 中添加 `content_type` 字段，使用 `IsIn` 验证器确保值合法

**理由：**
- 统一的数据验证策略
- 使用 class-validator 的 `IsIn` 确保只接受 'rich-text' 或 'markdown'
- 在 Swagger 文档中自动生成字段说明

**替代方案：**
- 使用自定义验证器：复杂度较高，`IsIn` 已经足够
- 后端不做验证：无法保证数据一致性，容易产生脏数据

## 风险 / 权衡

### 风险：数据库迁移可能失败
- **缓解措施：** 在开发环境充分测试迁移脚本，确保 ALTER TABLE 操作不会影响现有数据
- **回滚策略：** 使用数据库备份，如果迁移出现问题可以快速恢复

### 权衡：字段长度选择
- VARCHAR(20) vs VARCHAR(50)：选择 VARCHAR(20) 更节省空间，但预留扩展空间有限
- **权衡：** 当前只需 10-11 字符，VARCHAR(20) 已足够，未来格式名称变化不大

### 权衡：是否支持 NULL
- **决策：** 不支持 NULL，使用默认值 'markdown'
- **权衡：** 简化前端逻辑，但失去"未设置"状态的语义

### 风险：现有数据默认值可能不准确
- **缓解措施：** 提供数据库迁移脚本，允许管理员批量更新现有数据的格式类型
- **建议：** 在 API 文档中说明新字段的用途，引导用户正确设置格式
