# 文章内容类型管理规范

## 新增需求

### 需求:文章必须有默认的内容类型
系统必须为所有新创建的文章设置默认的内容类型为 `'rich-text'`。

#### 场景:创建新文章时自动设置默认类型
- **当** 用户创建新文章且未明确指定 `content_type` 参数
- **那么** 系统必须将 `content_type` 设置为 `'rich-text'`

#### 场景:创建新文章时明确指定类型
- **当** 用户创建新文章并明确指定 `content_type` 为 `'markdown'` 或 `'rich-text'`
- **那么** 系统必须使用用户指定的类型

### 需求:内容类型值必须有效
系统必须验证 `content_type` 字段的值只能是 `'markdown'` 或 `'rich-text'`。

#### 场景:无效的内容类型值
- **当** 用户提交 `content_type` 为无效值（如 `'html'`、`'pdf'` 等）
- **那么** 系统必须拒绝请求并返回验证错误

### 需求:API 文档必须反映正确的默认值
Swagger API 文档必须在 `CreatePostDto` 中正确显示 `content_type` 字段的默认值为 `'rich-text'`。

#### 场景:查看 API 文档
- **当** 开发者查看 Swagger 文档中的创建文章接口
- **那么** `content_type` 字段的默认值必须显示为 `'rich-text'`

### 需求:数据库默认值必须与业务逻辑一致
数据库表的 `content_type` 列默认值必须设置为 `'rich-text'`。

#### 场景:数据库层面默认值
- **当** TypeORM 自动生成表结构或在表中插入新记录
- **那么** `content_type` 列必须使用 `'rich-text'` 作为默认值

### 需求:历史数据应该得到修正
系统应该提供机制来修正历史数据中错误的 `content_type` 值。

#### 场景:数据迁移脚本
- **当** 运行数据迁移脚本
- **那么** 系统应该将所有 `content_type` 为 `'markdown'` 且内容格式实际为富文本的记录更新为 `'rich-text'`

#### 场景:手动数据修正
- **当** 管理员需要修正历史数据
- **那么** 系统应该提供管理界面或 API 来批量更新文章的内容类型
